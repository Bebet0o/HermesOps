# HermesOps Console Foundation

Status: **implemented by milestone 2P**

## Purpose

The HermesOps Console is the dedicated browser client for the HermesOps
Controller. Milestone 2P installs only its technical foundation: a deterministic
static build, a hardened loopback HTTP service, the product navigation shell,
and bounded probes and tests.

The Console remains an unprivileged client. It never opens SQLite, project
workspaces, the Hermes Agent endpoint, Docker sockets, or host paths.

## Initial deployment

| Component | Binding | Role |
| --- | --- | --- |
| Hermes Agent | `127.0.0.1:8642` | Upstream AI gateway |
| Controller API | `127.0.0.1:8765` | Authoritative control plane |
| Legacy Hermes WebUI | `127.0.0.1:8787` | Temporary direct Agent compatibility UI |
| HermesOps Console | `127.0.0.1:8788` | Dedicated HermesOps product UI |

The new Console does not replace or modify the legacy WebUI in 2P. It is
independent from that container and has no gateway credential or Agent network
path.

The final browser-to-Controller same-origin transport is introduced in later
Console milestones. In 2P, the Console intentionally performs no API or
WebSocket request. Its Content Security Policy therefore uses
`connect-src 'none'`.

## Dependency posture

The Console foundation has no Node.js, npm, pnpm, yarn, CDN, third-party script,
font, image, or runtime dependency. The committed distribution is generated by
`scripts/hermesops-console-build.py` using Python 3.11 standard library only.

The build is deterministic:

```bash
python3 scripts/hermesops-console-build.py check \
  --source console/src \
  --expected console/dist
```

The exact source file set is closed. Symlinks, hard-linked source files,
unexpected files, invalid UTF-8, NUL bytes, empty files, and oversized files are
rejected before output is replaced.

## Routes

The foundation serves one shell for these product routes:

- `/dashboard`;
- `/projects`;
- `/objectives`;
- `/executions`;
- `/reviews`;
- `/events`;
- `/administration`.

Each route states which later milestone will make it functional. The shell does
not simulate Controller data or report optimistic state.

The only service endpoints are:

- `GET|HEAD /health`;
- `GET|HEAD /version`.

No `/api` proxy exists in 2P.

## HTTP safety

The service:

- binds only to a validated loopback IP;
- serves an exact allowlist of routes and assets;
- never performs directory listing or generic filesystem resolution;
- rejects encoded paths, query strings, backslashes, invalid Host headers, and
  unsupported methods;
- limits concurrent request threads;
- applies a connection timeout;
- emits bounded problem documents without paths or exceptions;
- sends CSP, CORP, COOP, referrer, framing, MIME-sniffing, permissions, cache,
  and request-ID headers;
- suppresses the Python implementation version.

## Service operations

```bash
systemctl --user status hermesops-console.service
curl --fail http://127.0.0.1:8788/health
python3 scripts/hermesops-console-probe.py \
  --base-url http://127.0.0.1:8788
```

The service is installed as a user unit and is independent of the Controller,
orchestrator, supervisor, notifier, Hermes Agent, and legacy WebUI processes.

## Explicit 2P non-goals

Milestone 2P does not implement:

- browser login or logout integration;
- Controller API calls;
- CSRF, idempotency, or `If-Match` use in the frontend;
- WebSocket connectivity;
- project, Hermesfile, objective, review, or Recovery mutations;
- operational dashboard data;
- browser persistence;
- service-worker or offline command queues;
- replacement or reconfiguration of the legacy Hermes WebUI.
